# Quickstart: your first monitored table in 10 minutes

This is a start-to-finish walkthrough for someone opening DQX Studio for the
first time. By the end you'll have a real table being watched for quality
problems, with a rule applied, run, and results on screen — no rule-writing by
hand, no code.

Follow the steps in order. Each one builds on the last.

<Admonition type="note" title="Before you start">
You'll need a table in Unity Catalog that you already have access to. DQX Studio
only ever shows you data you can already see, so pick a table you know — an
`orders`, `customers`, or similar table works well.
</Admonition>

## 1. Open DQX Studio

Open the app. The **home** view greets you with an at-a-glance summary — how many
rules, tables, and collections you have, your overall data quality score, and a
trend of average quality over time. On a fresh setup these will be empty. That's
fine; you're about to change that.

## 2. Get a rule without writing one

A **rule** is a reusable definition of a data-quality expectation — for example,
"this column is never blank". Instead of writing one from scratch, let DQX Studio
suggest rules for you by **profiling** a table: it scans the data, looks at its
shape, and proposes sensible candidate rules.

Start a new profiling session, pick the table you chose above, and let it scan.
In a moment you'll see suggested rules — things like "this ID is never blank" or
"this amount is never negative". Accept one that looks useful. It lands in your
**Rules Registry** as a draft, ready to apply.

<Admonition type="tip" title="Prefer to describe it in words?">
You can also **generate a rule with AI** — type what you want in plain English
("email addresses should look valid") and the app proposes a check. Both aids
live on the [Create a rule](/docs/studio/authoring/create-a-rule) page.
</Admonition>

## 3. Apply the rule to your table

Now attach the rule to real data. Open your table's **Apply rules** view, find
the column the rule fits, and apply the rule to it. This turns the rule into a
**check** — the same rule can become many checks across different columns and
tables.

<ThemedImage
  alt="A monitored table's Apply Rules view, showing rules grouped by column with severity indicators and pass-threshold badges"
  sources={{
    light: useBaseUrl('/img/studio/table_apply_rules_light.png'),
    dark: useBaseUrl('/img/studio/table_apply_rules_dark.png'),
  }}
/>

While you're here, set two things for the check:

- **Severity** — how much a failure matters: **Low**, **Medium**, **High**, or
  **Critical**. It's a priority for triage and reporting, not a pass/fail switch.
- **Pass threshold** — optionally require that, say, "70% of rows must pass"
  before the check counts as healthy. This is handy for gradual clean-up. See
  [Set quality thresholds](/docs/studio/monitoring/quality-thresholds).

Your table is now a **monitored table** — the single place to see and manage that
table's data quality.

## 4. Let it go through approval

Rules don't go live the instant you make them. Your new rule is submitted for
review, and an approver signs off before it becomes active. This "four-eyes" step
means at least two people touch every rule before it affects a run.

For this walkthrough, assume it's approved. (If you're an approver or admin, you
can approve it yourself.) See the
[Approval workflow](/docs/studio/governance/approval-workflow) for how this works.

## 5. Run the checks

With an active check in place, run it. From your monitored table, trigger a run.
DQX Studio reads the table and evaluates the check against every row — producing
one **test** per row. It never changes your data; it only reports what it finds.
See [Run checks](/docs/studio/running/run-checks).

## 6. See your results

When the run finishes, open the table's **Results**. You'll see an overall **data
quality score**, a breakdown by dimension and severity, and — the useful part —
you can drill into the exact **rows that failed** so you know what to fix.

<ThemedImage
  alt="A Results view showing an overall data quality score, a quality trend over time, and a breakdown by dimension and severity"
  sources={{
    light: useBaseUrl('/img/studio/collection_results_light.png'),
    dark: useBaseUrl('/img/studio/collection_results_dark.png'),
  }}
/>

That's the whole loop: a rule, applied to a table, run, with results you can act
on. See [Measure quality health](/docs/studio/running/measure-quality-health) to
read a score, and [Fix a failing rule](/docs/studio/running/fix-a-failing-rule)
to work through the rows that didn't pass.

<Admonition type="tip" title="Failing rows are safe to inspect">
A failing check never deletes or edits your data. Failing rows are sampled into a
view purely so you can triage them.
</Admonition>
