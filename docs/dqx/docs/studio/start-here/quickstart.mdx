# Quickstart: your first monitored table in 10 minutes

This is a start-to-finish walkthrough for someone opening DQX Studio for the
first time. By the end you'll have a rule you built, applied to a real table,
run, with results on screen — no code.

Follow the steps in order. Each one builds on the last.

<Admonition type="note" title="Before you start">
You'll need a table in Unity Catalog that you already have access to. DQX Studio
only ever shows you data you can already see, so pick a table you know — an
`orders`, `customers`, or similar table works well.
</Admonition>

## 1. Create a reusable rule

A **rule** is a reusable definition of a data-quality expectation — for example,
"this column is never blank". Author it once here, and you'll be able to apply it
to as many tables as you like.

Go to **Rules** and choose **New rule**. You'll fill in two tabs:

- On the **About** tab, give the rule a **name** and a short **description**, pick
  a **default severity** (**Low**, **Medium**, **High**, or **Critical**), and
  choose a **dimension** (such as completeness or validity — the quality aspect
  this rule measures).
- On the **Implementation** tab, pick a check from **Basic Rules** — for example
  **is not null**, which flags any blank value. That's all a simple rule needs.

    dark: useBaseUrl('/img/studio/create_rule_basic_checks_dark.png'),

<Admonition type="tip" title="Prefer to describe it in words?">
On the About tab you can also describe the rule in plain English and let **AI**
generate the check for you. See [Create a rule](/docs/studio/authoring/create-a-rule)
for both approaches.
</Admonition>

Save the rule. It lands in your **Rules Registry** — your library of reusable
rules — ready to apply.

## 2. Add a table to monitor

Before you can apply a rule to a table, you register the table with DQX Studio.

Go to **Tables** and choose **Monitor a table**. Browse Unity Catalog and pick the
table you chose above. Because DQX Studio only shows data you already have access
to, if a table isn't listed it's a permissions matter, not a bug.

  alt="The Monitored Tables list, showing catalog, schema, and table with rule counts, DQ score, and last run"

## 3. Apply your rule to the table

Open your table and go to its **Apply Rules** tab. Find the column your rule fits
and apply the rule to it. This turns the rule into a **check** — the same rule can
become many checks across different columns and tables.

<ThemedImage
  alt="A monitored table's Apply Rules tab, showing rules grouped by column with severity indicators and pass-threshold badges"
  sources={{
    light: useBaseUrl('/img/studio/table_apply_rules_light.png'),
    dark: useBaseUrl('/img/studio/table_apply_rules_dark.png'),
  }}
/>

Here you can override the rule's defaults for this table:

- **Severity** — keep the rule's default, or change it for this check. Severity is
  a priority for triage and reporting; it doesn't decide pass or fail.
- **Pass threshold** — optionally require that, say, "70% of rows must pass" before
  the check counts as healthy. Thresholds are set per rule and per column, which is
  handy for gradual clean-up. See
  [Set quality thresholds](/docs/studio/monitoring/quality-thresholds).

Your table is now a **monitored table** — the single place to see and manage that
table's data quality.

## 4. Let it go through approval

Rules don't go live the instant you make them: your check is submitted for review,
and an approver signs off before it becomes active. See the
[Approval workflow](/docs/studio/governance/approval-workflow). For this
walkthrough, assume it's approved.

## 5. Run the checks

With an active check in place, trigger a run from your monitored table. DQX Studio
reads the table and evaluates the check against every row. It never changes your
data; it only reports what it finds. See [Run checks & schedule](/docs/studio/running/run-and-schedule).

## 6. See your results

When the run finishes, open the table's **Results**. You'll see an overall **data
quality score** and a breakdown of what passed and what didn't. The useful part is
the **Failed records** table — hover a failing row to see exactly which checks it
failed, each with its severity and the reason, so you know what to fix.

<ThemedImage
  alt="A Results view — hovering a failing row shows a popup listing the checks it failed with severity and reason"
  sources={{
    light: useBaseUrl('/img/studio/failed_record_popup_light.png'),
    dark: useBaseUrl('/img/studio/failed_record_popup_dark.png'),
  }}
/>

That's the whole loop: a rule, applied to a table, run, with results you can act
on. See the [Results page](/docs/studio/running/results) to read a score and work
through the rows that didn't pass.

<Admonition type="tip" title="Failing rows are safe to inspect">
A failing check never deletes or edits your data. Failing rows are sampled into a
view purely so you can triage them.
</Admonition>
