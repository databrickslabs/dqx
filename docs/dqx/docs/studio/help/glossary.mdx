# Glossary

Plain-language definitions of the terms you'll meet in DQX Studio, in alphabetical order. For how these ideas fit together, see [Key concepts](/docs/studio/start-here/key-concepts).

| Term | Meaning |
|---|---|
| **Admin** | A role that can do everything, including change workspace settings; see [Roles and permissions](/docs/studio/governance/roles-and-permissions). |
| **AI generation** | Describe the rule you want in plain English and the app proposes a check for you to review; see [Generate with AI](/docs/studio/authoring/generate-with-ai). |
| **Approval (four-eyes)** | The safeguard that a draft must be reviewed and approved by someone other than its author before it goes live; see [Approval workflow](/docs/studio/governance/approval-workflow). |
| **Approver (Rule Approver)** | A role that can approve or reject anyone's drafts, not just their own. |
| **Ask Genie** | A box on Results views where you can ask questions about your quality in plain English and get an answer back. |
| **Audit trail** | The record of every rule, role, and schedule change: who made it, when, and what it was before and after. |
| **Author (Rule Author)** | A role that can create and edit its own drafts, but not approve them. |
| **Check** | A rule applied to a specific column or table (one rule can become many checks across your data). |
| **Collection (data product)** | A group of related monitored tables you view, run, and schedule together, so you can see quality across a whole data product at once; see [Collections](/docs/studio/monitoring/collections). |
| **Data contract (ODCS)** | An Open Data Contract Standard file describing a dataset's expectations, which you can import to create draft rules; see [Import rules](/docs/studio/authoring/import-rules). |
| **DQ score** | A single percentage summarizing how much of your data passed its checks in a run. |
| **Draft** | A rule that has been created but is not yet live; it stays a draft until it is approved. |
| **Dry run (Test)** | Trying a rule against a few sample rows before you save it, which changes and stores nothing; see [Create a rule](/docs/studio/authoring/create-a-rule). |
| **Error (severity)** | A check whose failure fails the run, marking that data as invalid. |
| **Governed tag** | A tag you attach to a rule (for example `class.location`) so the app can suggest which columns it applies to; see [Governed tags](/docs/studio/monitoring/governed-tags). |
| **Monitored table** | A Unity Catalog table you've applied rules to, and the single source of truth for that table's data quality; see [Monitor a table](/docs/studio/monitoring/monitor-a-table). |
| **On-behalf-of** | The principle that you only ever see catalogs and tables you already have access to, because profiling and tests run as you. |
| **Pass threshold** | A minimum share of rows a check must pass to count as healthy (for example 70%), shown as a badge like `< 70%`; see [Quality thresholds](/docs/studio/monitoring/quality-thresholds). |
| **Profiling** | Scanning a table so the app can suggest candidate rules based on the shape of the data; see [Generate by profiling](/docs/studio/authoring/generate-by-profiling). |
| **Quality dimension** | The category a check measures quality against: completeness, validity, accuracy, consistency, uniqueness, or timeliness. |
| **Quarantine** | A view holding a sample of the rows that failed a check, so you can look into them; your data itself is never changed. |
| **Retention** | How long the app keeps run history and metrics, and separately how long it keeps quarantined sample rows; set by an admin in [Governance controls](/docs/studio/governance/governance-controls). |
| **Rule** | A reusable definition of a data-quality expectation, such as "email is not null." |
| **Rules Registry** | The library of reusable rule definitions you author once and reuse across many tables; see [Key concepts](/docs/studio/start-here/key-concepts). |
| **Run (manual / scheduled)** | One evaluation of a table's or collection's checks, started by you (manual) or on a set schedule; see [Run checks](/docs/studio/running/run-checks). |
| **Runner** | An added privilege that lets someone trigger runs, separate from their role; admins can always run. |
| **Test** | A single evaluation of one check against one row (or aggregate) during a run, so a check over a million rows performs a million tests. |
| **Unity Catalog** | Databricks' catalog of your tables and their permissions, which the app reads to know what data you can see. |
| **Viewer** | A read-only role that can see rules and results but not change them. |
| **Warning (severity)** | A check whose failure is reported but does not fail the run. |

<Admonition type="tip" title="Not sure why a table is missing?">
If a catalog or table you expect isn't listed, it's almost always a permissions matter, not a bug: you only see data you already have access to in Unity Catalog. See [Data access and security](/docs/studio/governance/data-access-and-security).
</Admonition>
