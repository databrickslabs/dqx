# Troubleshooting & FAQ

Short answers to the questions stewards ask most. If something looks wrong, start here — most surprises turn out to be permissions or a setting, not a broken tool.

### Why isn't my table listed?

DQX Studio only shows you catalogs and tables you already have permission to in Unity Catalog. If a table is missing, it almost always means your account can't see it yet — not that the app is broken. Ask your workspace admin to grant you access to the catalog or schema, then it will appear.

See [Data access and security](/docs/studio/governance/data-access-and-security).

### Why can't I approve my own rule?

DQX Studio uses four-eyes approval: the person who writes a rule can't be the one who approves it, so a second pair of eyes always signs off. If you're a Rule Approver or Admin you can approve other people's drafts, but not your own. This keeps changes accountable.

See [Approval workflow](/docs/studio/governance/approval-workflow).

### If a rule fails, does my data get deleted or changed?

No. DQX Studio reads and evaluates your data — it never modifies, deletes, or moves it. When rows fail a check, a small sample is copied into a separate view so you can inspect what went wrong. Your source table is untouched.

See [Fix a failing rule](/docs/studio/running/fix-a-failing-rule).

### What's the difference between an error and a warning?

An **error** fails the run — it's a problem you want to block on. A **warning** is reported but doesn't fail the run — it's something to keep an eye on. You set a default when you write a rule and can change it per table.

See [Key concepts](/docs/studio/start-here/key-concepts).

### What's the difference between a rule, a check, and a test?

A **rule** is a reusable definition of an expectation ("email is not null"). A **check** is that rule applied to a specific table and column. A **test** is a single evaluation of a check against one row during a run — so one check over a million rows runs a million tests.

See [Rules, checks, and tests](/docs/studio/start-here/key-concepts#rules-checks-and-tests).

### My check is always red — is that bad?

Not necessarily. A check can require that a percentage of rows pass before it turns green (shown as a badge like `< 70%`). If yours is always red, the threshold may be set too strict for the current state of the data. Adjust it to a realistic level and raise it over time as quality improves.

See [Quality thresholds](/docs/studio/monitoring/quality-thresholds).

### Do I need a collection?

No — collections are optional. Use one when you have several related tables (for example, the tables behind a data product or a Genie space) that you want to view, run, and schedule together. Until then, you can monitor tables on their own.

See [Collections](/docs/studio/monitoring/collections).

### Which authoring option should I use — form or SQL?

If you're not writing code, use the form-based **Basic Rules** option. It walks you through the check with plain fields and no SQL required. The SQL option exists for advanced users who want to write an expression by hand.

See [Create a rule](/docs/studio/authoring/create-a-rule).

### How do I know which role I have?

Open your **Profile** page — it shows your current role (Viewer, Rule Author, Rule Approver, or Admin) and whether you can trigger runs. If you need a different role, contact your workspace admin.

See [Roles and permissions](/docs/studio/governance/roles-and-permissions).

### A run failed to start, or a table couldn't be submitted

This is usually one of two things: you don't have permission to the table or compute, or the rule has a problem (for example, a column that doesn't exist). Read the failure message shown on screen — it names the cause. Permissions issues are the most common.

See [Data access and security](/docs/studio/governance/data-access-and-security).

### Can I get my rules out of DQX Studio?

Yes. Rules can be downloaded and exported as YAML, so you always keep a portable copy. The same YAML format is what the app reads back in.

See [Import rules](/docs/studio/authoring/import-rules) for the format.

### Where do I get more help?

For questions about DQX Studio, or to report a problem, open an issue on the [DQX GitHub repository](https://github.com/databrickslabs/dqx). For code-level details on the underlying data-quality engine, see the [DQX Core documentation](/docs/guide/).
