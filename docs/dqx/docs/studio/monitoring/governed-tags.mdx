# Governed tags & auto-apply

This page is for **Data Stewards** and **Data Governance Managers** who want the
same class of data — all location fields, all email addresses, all personal
information — to get the same quality rules everywhere, automatically, without
attaching rules to columns table by table.

A **governed tag** is a label your organization agrees on for a kind of data —
for example `class.location`, `class.email`, or `class.pii`. The idea is simple:
instead of pointing a rule at a named column, you point it at a **tag**. Any
column that carries that tag then gets the rule applied to it — automatically,
wherever that column lives.

## How it works

The key move happens when you **author a rule**. On the rule's column, instead of
leaving it as a plain placeholder, you attach a governed tag to it using **+ Apply
to a tag**:

<ThemedImage
  alt="The rule editor's Columns used section, where a column can be applied to a governed tag"
  sources={{
    light: useBaseUrl('/img/studio/create_rule_implementation_light.png'),
    dark: useBaseUrl('/img/studio/create_rule_implementation_dark.png'),
  }}
/>

Now the rule isn't tied to a column *name* — it's tied to a *tag*. From then on,
**any table column that carries that governed tag has the rule applied to it
automatically.** Tag a column `class.email` on a new table, and every rule scoped
to `class.email` starts checking it — no manual mapping, on that table or any
other.

<Admonition type="tip" title="Tag the column, not the rule">
Governed tags attach to the **column** inside a rule, not to the rule as a whole.
That's what lets the rule follow the tag onto real columns wherever they appear.
</Admonition>

## Why this matters

Without governed tags, someone has to remember that an email lives in a column
called `email` on one table, `email_address` on another, and `contact_email` on a
third — and hand-apply the email-format rule to each one. That's slow, and it's
how a column quietly slips through unchecked.

Governed tags flip it around. You express *what kind of data* a rule is for,
once. Any column tagged as that kind of data — today or in the future — is
covered automatically. Your standards stay consistent across every table, and new
tables inherit the right rules the moment their columns are tagged.

## A worked example

Say you have a rule, **"is a valid ISO country code"**, and on its column you
apply the tag `class.location`.

Now a customer table has a `country_code` column tagged `class.location` in Unity
Catalog. You can see that tag on the column in the table's schema:

<ThemedImage
  alt="Monitored table schema showing a class.location governed tag on the country_code column"
  sources={{
    light: useBaseUrl('/img/studio/table_detail_thumb_light.png'),
    dark: useBaseUrl('/img/studio/table_detail_thumb_dark.png'),

Because the column carries `class.location`, the country-code rule is applied to
`country_code` automatically — and to any other `class.location` column on any
other table. A **matched-tag chip** marks these checks so you can see at a glance
that they came from a tag rather than being mapped by hand.

## How to set it up

### 1. Define governed tags (admin)

An admin curates the approved set of tags in **Admin Settings → Tags**. This is
the shared taxonomy everyone else draws from.

<ThemedImage
  alt="Admin Settings Tags screen listing governed tag definitions"
  sources={{
    light: useBaseUrl('/img/studio/settings_tags_light.png'),
    dark: useBaseUrl('/img/studio/settings_tags_dark.png'),
  }}
/>

See [Set up DQX Studio for your organization](/docs/studio/start-here/set-up-for-your-org)
for the wider admin setup.

### 2. Apply a tag to a rule's column (author)

When you create or edit a rule, select its column and use **+ Apply to a tag** to
attach a governed tag — for example, tag the column in your "is a valid ISO
country code" rule with `class.location`. You pick from the tags your admin
defined, so the taxonomy stays consistent. See
[Create a rule](/docs/studio/authoring/create-a-rule).

### 3. Tag your columns in Unity Catalog

For a rule to land, the target columns need the matching governed tag in Unity
Catalog. Once a column is tagged, every rule scoped to that tag applies to it
automatically — you'll see the resulting checks (with a matched-tag chip) on the
table's **Apply Rules** tab. See
[Monitor a table](/docs/studio/monitoring/monitor-a-table).

<Admonition type="note" title="Matching relies on Unity Catalog tags">
Auto-apply works off the governed tags set on your columns in Unity Catalog. If a
rule isn't landing where you expect, check that the column actually carries the
matching tag.
</Admonition>
