# Assign rules

Assigning rules is where the reusable definitions in your
[Rules Registry](/docs/studio/start-here/key-concepts) start doing work: you
attach them to a [monitored table](/docs/studio/monitoring/)'s columns and tune
them for that table. Each rule you apply to a column becomes a **check** — a
single rule can become many checks across your tables and columns.

Open the **Apply Rules** tab on a monitored table to get started.

  alt="Apply Rules tab showing rules mapped to columns, severity indicators, pass-threshold badges, and a Suggest rules button"
  sources={{

    dark: useBaseUrl('/img/studio/table_apply_rules_dark.png'),

/>

You can look at your checks two ways:

- **By column** — see every check attached to a given column.
- **By rule** — see every column a given rule is checking.

For each check you apply, you can tune it for this table without touching the
underlying rule:

- **Override the severity.** Set how much a failure matters on this table —
  **Low**, **Medium**, **High**, or **Critical**. The rule carries a default
  severity; you can override it here. Severity is a priority for triage and
  reporting, not what decides pass or fail. See
  [Key concepts](/docs/studio/start-here/key-concepts#severity).
- **Set a pass threshold.** Instead of demanding every row pass, you can require,
  say, 70% of rows to pass for the check to count as healthy. This is what decides
  whether a check is green. Useful when you're improving quality gradually. See
  [Quality thresholds](/docs/studio/monitoring/quality-thresholds).

## Pin a rule version

Rules in the registry evolve — an author edits a rule and it gets a new version.
Each check on your table shows a **version selector** (for example, *Latest
(v1)*). You choose how that check tracks the rule:

- **Latest** — the check always uses the newest approved version of the rule, so
  improvements roll out to your table automatically.
- **Pin to a version** — lock the check to a specific version, so it stays exactly
  as it is until you deliberately move it. Pin when you want a table's behavior to
  stay stable regardless of registry changes.

Pinning is per check, so you can let most checks follow the latest while holding a
sensitive one steady.

## Suggest rules with AI

You don't have to map every check by hand. Select **Suggest rules** and DQX
Studio looks at this specific table — its columns, their names and types, and the
shape of the data — and proposes a tailored set of checks for it: an `id` that
should be unique, an `email` that should match a pattern, an `amount` that
shouldn't be negative, and so on.

<ThemedImage
  alt="Suggest rules with AI: the app proposes a tailored set of checks for a table's columns"
  sources={{
    light: useBaseUrl('/img/studio/feat_suggest_light.png'),
    dark: useBaseUrl('/img/studio/feat_suggest_dark.png'),

/>

It's the fastest way to go from an unmonitored table to a sensible starting set
of checks. You stay in control — review each suggestion, keep the ones that fit,
drop the rest, and adjust severities or thresholds before you submit. Think of it
as a knowledgeable first draft of "what should we be checking on this table?",
not a decision made for you.

<Admonition type="note" title="Governed tags apply rules automatically">
If a rule's column is marked with a governed tag (for example `class.location`),
that rule is applied automatically to any column on this table carrying the same
tag — so tagged rules land on the right columns without you mapping them by hand.
See [Governed tags](/docs/studio/authoring/governed-tags).
</Admonition>

<Admonition type="info" title="Your data is never changed">
Applying rules and running checks only reads and evaluates your data. A failing
check never deletes or modifies the table — failing rows are sampled into a
quarantine view for triage.
</Admonition>

## Save as draft, or submit for review

When you've applied the checks you want, you can **save as a draft** to keep
working, or **submit for review**. Once your changes are approved, the table is
actively monitored and its checks run on the schedule set for it.

- Learn how review and approval works in the
  [Approval workflow](/docs/studio/governance/approval-workflow).
- Learn how and when checks run in [Run checks & schedule](/docs/studio/running/run-and-schedule).
