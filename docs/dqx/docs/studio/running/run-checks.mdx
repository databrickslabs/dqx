# Run checks

You've applied rules to a table. Now you want to see how the data is doing. A **run** evaluates every check on a table (or a whole collection) against the current data and records the results, so you can spot problems and track quality over time.

There are two ways to run checks: on demand, or on a schedule.

## Run now (manual)

Open a monitored table or a collection and click **Run now**. The run evaluates all of that table's or collection's checks against the data and records the results.

You don't have to wait around. The run continues in the background, so you can leave the page or go do something else. When it finishes, it shows up in **Runs History**.

<Admonition type="note" title="You need the Runner privilege">
Triggering a run needs the **Runner** privilege. Admins have it automatically. If you don't see **Run now**, ask an admin to grant it — see [Roles and permissions](/docs/studio/governance/roles-and-permissions).
</Admonition>

## Put it on a schedule (automatic)

For quality you want to keep an eye on, put a collection (or a single table) on a schedule so it runs itself. Choose the cadence that fits:

- **Hourly**
- **Daily** at a time you pick
- **Weekly** on the days you choose
- **Custom** — a cron expression, if you need something more specific

You can **pause**, **resume**, or **delete** a schedule at any time, right where you set it up.

<Admonition type="tip" title="Times use your workspace timezone">
Scheduled times follow the workspace timezone set by your admin. If a scheduled run is still going when the next one is due, the app skips the new one rather than running two at once.
</Admonition>

## See the results

After a run, results appear in two places:

- **Runs History** — every past run, with its status and how long it took.
- **Results views** — the quality score, trends, and failing rows for a table or collection. See [Measure quality health](/docs/studio/running/measure-quality-health).

<Admonition type="info" title="One run, many tests">
A single run of one check over your data produces many **tests** — one per row (or aggregate) it evaluates. So a run over a million rows can produce a million tests. That's why you'll see counts like "421,121 failed of 2,350,000 tests". See [Rules, checks, and tests](/docs/studio/start-here/key-concepts#rules-checks-and-tests).
</Admonition>
