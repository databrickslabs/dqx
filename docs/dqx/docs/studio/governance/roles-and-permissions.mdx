# Roles & permissions

This page is for anyone who wants to know what they can do in DQX Studio, and for admins who decide what everyone else can do. Your role controls whether you can view rules, write them, approve them, or change settings.

## Which role am I, and what can I do?

Find yourself below, then read across.

- **I just want to see the rules and results.** You're a **Viewer**. You can look at everything but change nothing.
- **I write and edit rules.** You're a **Rule Author**. You can create rules and edit your own drafts, but someone else approves them.
- **I sign off on other people's work.** You're a **Rule Approver**. You can approve or reject drafts from anyone.
- **I run the whole thing and change settings.** You're an **Admin**. You can do everything, including admin settings.

Not sure which one you are? Your [Profile page](#how-to-find-your-role) tells you exactly.

## The four roles

Every person has one primary role. Higher roles include everything the roles below them can do.

| Task | Viewer | Rule Author | Rule Approver | Admin |
|---|:---:|:---:|:---:|:---:|
| View rules, tables, collections, and results | ✅ | ✅ | ✅ | ✅ |
| Create rules and edit your own drafts | | ✅ | ✅ | ✅ |
| Submit drafts for approval | | ✅ | ✅ | ✅ |
| Approve or reject anyone's drafts | | | ✅ | ✅ |
| Change admin settings (entitlements, retention, tags) | | | | ✅ |

## The Runner privilege (this one is separate)

Running checks is controlled on its own, **in addition to** your role. It is not a fifth rank above Admin.

<Admonition type="caution" title="Runner is additive, not a rank">
The <strong>Runner</strong> privilege lets you trigger a run against a table or collection. It sits alongside your role rather than above it. A <strong>Viewer with Runner</strong> can run checks but still cannot author rules. A <strong>Rule Author without Runner</strong> can write rules but cannot run them. Admins are always runners automatically.
</Admonition>

So when you look at what someone can do, read two things: their **role** (view / author / approve / admin) and whether they **also have Runner** (can trigger runs).

## Keeping authoring and approval separate

DQX Studio uses **four-eyes approval**: the person who writes a rule can't be the only person who approves it. If you're a Rule Author, you submit your draft and someone with approver rights reviews it before it goes live. This keeps a second set of eyes on every change.

An approver or admin who writes a rule still can't approve their own draft in the same step — the review is meant to be independent. See [Approval workflow](/docs/studio/governance/approval-workflow) for the full draft → submit → review → approve → active path.

## How to find your role

Open the menu in the top-right corner and go to your **Profile** page. It shows:

- Your **role** (Viewer, Rule Author, Rule Approver, or Admin).
- Whether you have the **Runner** privilege.
- Which **workspace groups** mapped you to that role, so you know where your access comes from.

## How roles are assigned

You don't get a role assigned to you by name. Instead, an admin maps your **Databricks workspace groups** to DQX Studio roles. If you belong to a group that's mapped to Rule Author, you're a Rule Author.

Admins do this in **Admin Settings → Entitlements**.
