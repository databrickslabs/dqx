## Redesigned app & new documentation

- **A ground-up app redesign.** A cleaner layout and consistent patterns across
  authoring, monitoring, results, and governance.
- **Substantially faster and more fluid.** Lists, tables, and results load
  quicker and navigation feels noticeably smoother.
- **New documentation.** DQX is now documented as two tracks —
  [DQX Core](/docs/guide/) (the Python package) and
  [DQX Studio](/docs/studio/) (the no-code app) — with a task-oriented Studio
  guide illustrated with real screenshots.

## Authoring

- **[Rules Registry](/docs/studio/start-here/key-concepts).** Build a library of
  reusable, versioned rule definitions once and apply them across as many tables
  as you like. Each rule can become many checks.
- **[Low-code rule builder](/docs/studio/authoring/create-a-rule).** Author rules
  in a visual form — pick a built-in check, or write a custom condition with a
  plain-language condition builder, AND/OR chaining, filters, and joins — no code
  required.
- **[Suggest rules with AI](/docs/studio/monitoring/assign-rules#suggest-rules-with-ai).**
  Point the app at a table and it proposes a tailored set of checks for its
  columns and data, as a reviewable first draft you stay in control of.
- **[Profiling](/docs/studio/monitoring/profiling).** Scan a table to measure the
  shape of its data and get suggested candidate rules drawn from the statistics,
  staged for you to keep or discard.
- **[Severity & dimensions](/docs/studio/start-here/key-concepts#severity).** Give
  every rule a severity (a customizable Low / Medium / High / Critical priority
  for triage and reporting) and a quality dimension such as completeness or
  validity, so results can be grouped and ranked by what matters.
- **[Rule versioning](/docs/studio/monitoring/assign-rules#pin-a-rule-version).**
  Rules are versioned as they evolve, and each check on a table can either track
  the latest approved version automatically or be pinned to a specific version
  for stability.
- **[Import & export](/docs/studio/authoring/import-rules).** Import rules from
  DQX YAML or from ODCS data contracts, bulk-import across many tables at once,
  and export rules back out.

## Monitoring

- **[Monitored tables](/docs/studio/monitoring/).** Apply rules to a Unity Catalog
  table to make it a monitored table — the single source of truth for that
  table's data quality, with per-table severity and threshold overrides and bulk
  registration.
- **[Table overview & sample data](/docs/studio/monitoring/).** Each monitored
  table has an overview that shows its schema, a preview of sample data, and an
  ask-a-question box, so you can see what you're working with before and after
  applying rules.
- **[Quality thresholds](/docs/studio/monitoring/quality-thresholds).** Require a
  share of rows to pass (e.g. 95%) rather than demanding perfection, with
  per-rule and per-column overrides and clear breach indicators — ideal for
  improving quality gradually.
- **[Governed tags & auto-mapping](/docs/studio/authoring/governed-tags).** Tag a
  rule's column with a Unity Catalog governed tag and it applies automatically to
  any table column carrying that tag, with matched-tag chips showing where it
  landed.
- **[Collections (data products)](/docs/studio/monitoring/collections).** Group
  related monitored tables into a data product you can view, run, and schedule
  together — quality across related tables at a glance.

## Running & reviewing

- **[Runs & scheduling](/docs/studio/running/run-and-schedule).** Run checks on
  demand or on an hourly, daily, weekly, or custom schedule, with pause/resume
  and a full runs history.
- **[Results dashboard](/docs/studio/running/results).** Read the DQ score and
  trend, break results down by dimension, severity, rule, and column, and drill
  all the way to the failing records.
- **[Ask Genie](/docs/studio/running/genie).** Ask questions about your results
  in plain English and get an answer with a chart — no SQL or query building.
- **[Comments](/docs/studio/running/results#investigate-and-act-on-a-failure).**
  Discuss and hand off findings with a comment thread on results, each note
  keeping its author and timestamp.

## Governance & security

- **[Roles & entitlements](/docs/studio/governance/permissions-and-entitlements#entitlements).**
  Map Databricks workspace groups to DQX Studio roles — Viewer, Rule Author,
  Rule Approver, and Admin — so access follows people as they join and leave.
- **[Four-eyes approval](/docs/studio/governance/approval-workflow).** Changes
  move through a draft → submit → review → approve workflow, with a diff and
  comments, so a second person signs off before anything goes live.
- **[Object-level permissions](/docs/studio/governance/permissions-and-entitlements#permissions).**
  Grant View, Modify, Apply, and Execute on a specific rule, table, or
  collection, with sensible defaults seeded on create that you can narrow at any
  time.
- **[Data access respects Unity Catalog](/docs/studio/governance/).** The app
  browses and reads data as you, so you only ever see what you already have
  access to.

## Setup & administration

- **[Admin Settings](/docs/studio/governance/).** A single place to manage
  entitlements, governed tags, data retention, the AI endpoint, and the built-in
  demo.
- **[Built-in demo](/docs/demos#dqx-studio-built-in-demo).** Deploy a set of
  sample tables, rules, collections, and past runs to explore every feature
  against realistic data before pointing DQX Studio at your own tables.
