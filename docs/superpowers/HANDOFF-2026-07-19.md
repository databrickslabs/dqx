### What it does (your 7 asks, all delivered)
A check is now a **soft (warning) failure when fewer than X% of its rows pass**.
Resolution chain, first-non-null: **per-column override → per-rule override →
registry-rule default → admin default (fallback 70)**.

1. **Admin setting** "Default pass threshold" (70) — Settings → **Governance**
   tab (`DefaultPassThresholdSettings`, mirrors the draft-run-sample-limit card).
2. **Registry-rule default** — optional field in the rule form's **advanced
   (About) section**; persists in the rule's `user_metadata.pass_threshold`.
3. **By-rule apply view** — a `⚠ <pct>%` **pill beside the severity dropdown**
   (new shared `ThresholdPill`); click to override the rule-level default.
   Placeholder shows the effective default (registry ?? admin).
4. **By-column apply view** — the same pill per column, overriding further;
   persists as `user_metadata.column_pass_thresholds` (`{col: pct}`) on the
   applied rule.
5. **DQX-native?** — investigated and **rejected**: DQX core has no per-row
   pass-rate primitive (criticality is binary warn/error); the only native
   expression rewrites each row check as a dataset aggregate, which destroys
   per-row quarantine detail. So breach is evaluated **app-side at
   results-query time** from the per-check pass/fail counts each run already
   stores. **← the one architectural decision to sanity-check.**
6. **Results indicators** — a `⚠` breach glyph (amber=warn, red=error) in the
   drill-downs (dimension / severity / rule / column / table rows), in the
   run-picker dropdown entries, and as a per-run marker on the over-time trend
   chart.
7. **warn vs error** — yes: DQX `criticality` is threaded from the results view
   through to each aggregate's `breach_criticality`, so breaches show the right
   colour/severity.

### Key design facts
- **No DB migrations.** Admin default = a `dq_app_settings` k/v row; registry
  default = reserved `user_metadata` key (like dimension/severity); per-rule =
  the **existing** `dq_applied_rules.pass_threshold` column (previously
  persisted+materialized but never enforced — now repurposed); per-column = a
  reserved `user_metadata` map.
- **Breach math is exact integer arithmetic** `(total-failed)*100 < threshold*total`
  (strict `<`; total==0 never breaches) — a float version was caught in review
  producing false breaches at exact-tie thresholds on round row counts
  (10/20/50/100). Single helper `_breach_criticality`; every roll-up funnels
  through it.
- **Precedence single-source-of-truth:** backend `resolve_pass_threshold`
  (`registry_models.py`); the frontend pill placeholders reimplement the same
  order (nullish, so 0 is a real value everywhere).
